# 活动内容生产工作流蓝图：一张表进去，海报+推文+视频（可选）出来

## 文档定位

本文档定义一套可落地的 AI 辅助活动内容生产工作流，覆盖从「运营填表」到「海报+推文+视频定稿」的完整链路。视频生成为可选模块，团队按需启用。用于团队流程评审，评审通过后进入 Phased 实施。

**本文档的读者是负责落地的工程 Agent**：以下内容尽量给到节点级、字段级、格式级的确定性描述，减少实施时的自由发挥空间。凡是标注 `⟦实施约束⟧` 的地方为硬性要求，不可省略或简化。

**v3.2 关键更新（面向自部署落地 Agent）**：
- **新增第十章「自部署与实施细则」**——补齐"自己开发+自己部署"所需的全部工程细节：部署拓扑与**公网 HTTPS 入口（飞书回调能打进来的前提）**、服务器规格与 Docker 部署、n8n 生产环境变量（PostgreSQL/queue/encryption key 等）、飞书自建应用配置、密钥清单、服务间数据契约、安全基线、组件级+端到端测试用例、发布与运维。
- 前置清单 P15–P19 拆出自部署基建，总工时修订为 ~53h（多出 ~17h 一次性基建是"数据自主可控"的代价）。

**v3.1 关键更新（面向低代码团队）**：
- **海报渲染从「手写 Pillow/ImageMagick 脚本」改为「稿定开放平台可视化模板 + autofill API」**——设计师在稿定图形界面维护模板，改品牌/加版式零代码；n8n 仅用一个 HTTP 节点填数据出图。残留代码仅"隐式元数据写入"一处，一次性配置（详见第 3A-post 层）。
- 工具封装者的后处理开发量大幅下降，品牌迭代主体从"工具封装者改代码"转为"设计师改模板"。

**v3.0 关键更新（基于 v2.0 流程评审）**：
- **编排架构重构**：从「单体一张 n8n 大流程」改为「派发器 + 3 个独立异步子工作流 + 飞书状态看板」——一个产物失败不拖垮其他产物，可独立重跑（详见第 2 层）
- **新增失败/重试/幂等规范**：补齐 v2.0 缺失的异常处理维度（详见第 2.5 节）
- **合规标识从「查」改为「加 + 查」**：生成管线主动写入显式标识 + 隐式元数据（GB 45438-2025 强标要求），检查项退化为验证（详见第 2.6 节、第 3A-post 层）
- **审核分级重新校准**：概率性检测器（地图/敏感图/幻觉）从 🔴 自动驳回降为 🟡 强制人工复核，仅确定性检查保留 🔴（详见第三章）
- **新增人工反馈 → 再生成回环**：审核人的自然语言修改意见（"换配色""Tone 更活泼"）有明确回灌路径（详见第 3.8 节）
- **新增可观测性/运行台账**：每次 run 结果落库，支撑每月 Review（详见第 2.7 节）
- **底图色彩策略**：从「事后强行 remap」改为「生成端约束 + 仅校验不强改」（详见第 3A 层）
- **字体合规**：默认字体从微软雅黑改为思源黑体（开源可商用），规避字体版权风险（详见第四章）

## 一、工作流全景

```
运营填表 ──→ n8n 派发器 ──→ 异步触发 3 个独立子工作流（fire-and-forget）
  (1min)      (<10s,秒返)         │
                                  ├─→ ① 海报子工作流（自跑 ~2min）──→ 回写飞书「海报待审」
                                  │     AI出底图→稿定模板填充(含AI标识)→写隐式元数据→自动检查→回写
                                  │
                                  ├─→ ② 推文子工作流（自跑 ~3min）──→ 回写飞书「推文待审」
                                  │     Dify写推文→事实核查→自动检查→加AI标识→回写
                                  │
                                  └─→ ③ 视频子工作流（可选,自跑 ~5min）──→ 回写飞书「视频待审」
                                        即梦/可灵→模板固化品牌→自动检查→回写
                                                      │
              飞书多维表格 = 状态看板（三个产物状态各自独立更新）
                                                      │
                                                      ▼
                                    人工审核 ──→ 通过/修改/驳回 ──→ 定稿发布
                                     (5-15min)        │
                                                      └─ 修改：自然语言意见 → 再生成回环（第3.8节）
```

**核心设计原则**：
- **一次填表，多处复用**：同一份活动信息驱动海报、推文、视频的并行生成
- **AI 管创意，模板管品牌**：AI 负责氛围和创意，Logo/字体/色彩/版式由稿定可视化模板确定性保证（零渲染代码）
- **解耦独立**：三个产物拆成独立子工作流，各自独立运行、独立失败、独立重跑——一个挂掉不影响其余 ⟦实施约束⟧
- **主流程不等待**：派发器秒级返回，长任务在子工作流里自跑，状态回写飞书看板 ⟦实施约束⟧
- **自动检查 + 人工审核双层把关**：机器查已知规则，人负责判断和决策
- **确定性才阻断**：只有确定性高精度检查可设 🔴 自动驳回；概率性检测一律 🟡 强制人审 ⟦实施约束⟧
- **合规标识主动加**：AI 产物在生成阶段即写入显式+隐式标识，不依赖事后补 ⟦实施约束⟧
- **审核体系可扩展**：检查规则按类型注册，新增规则不影响已有流程

## 二、编排层（n8n）

### 2.1 架构总览：1 个派发器 + 3 个子工作流 + 1 个错误工作流

⟦实施约束⟧ 本流程在 n8n 中拆为 **5 个独立 workflow**，不得合并为单个大 workflow：

| Workflow | 角色 | 触发方式 | 运行时长 |
|----------|------|----------|----------|
| `WF-0 派发器` | 监听飞书→校验→幂等检查→异步派活→登记状态 | 飞书 Webhook（新增行） | < 10s（秒返） |
| `WF-1 海报子工作流` | 出底图→品牌后处理→检查→回写 | HTTP Webhook（被 WF-0 调用） | ~2min |
| `WF-2 推文子工作流` | Dify 写稿→核查→检查→回写 | HTTP Webhook（被 WF-0 调用） | ~3min |
| `WF-3 视频子工作流`（可选） | 视频生成→品牌固化→检查→回写 | HTTP Webhook（被 WF-0 调用） | ~5min |
| `WF-9 错误工作流` | 捕获任一 workflow 未处理异常→回写失败状态→告警 | n8n Error Trigger | 即时 |

**为什么用 HTTP Webhook 而不是 Execute Workflow 节点做异步**：
> ⟦实施约束⟧ n8n 的 `Execute Workflow` 节点在 `Wait For Sub-Workflow Completion = false` 时存在已知缺陷——父流程结束时会**中止（abort）尚未跑完的子流程**。因此真正的 fire-and-forget 必须这样做：每个子工作流以独立的 **Webhook 节点**为触发器；WF-0 用 **HTTP Request 节点**调用子工作流的 Webhook URL，子工作流的 Webhook 设为 **"Respond Immediately"（立即响应 200）**，WF-0 收到 200 即继续，子工作流在自己的 execution 里独立跑完。这样父子生命周期解耦，子流程不会被中止。

### 2.2 WF-0 派发器（节点级）

```
[节点1] 飞书 Webhook Trigger
  · 监听飞书多维表格「新增记录」事件
  · 输出：record_id + 全部字段
        ↓
[节点2] Set：生成 run_id 与 content_version
  · run_id = `${record_id}-${timestamp}`（全局唯一，贯穿全链路用于追踪和台账）
  · content_version = 1（再生成时 +1，见第 3.8 节）
        ↓
[节点3] 幂等检查（⟦实施约束⟧ 防重复生成）
  · 读取该 record_id 当前「海报状态/推文状态/视频状态」字段
  · IF 任一状态 ∈ {生成中, 待审, 已定稿} 且非「再生成」触发 → 停止，飞书提示"该行已在处理中"
  · ELSE 继续
        ↓
[节点4] 参数校验
  · 必填字段齐全？→ 缺失则飞书通知运营补填 + 状态置「校验失败」+ 停止
  · 模板名/风格名是否在已注册清单内？→ 不匹配则报错并停止
  · 「生成视频=是」时，视频三字段是否齐全？
        ↓
[节点5] 飞书回写：登记初始状态
  · 海报状态=生成中，推文状态=生成中，视频状态=（生成中 | 不适用）
  · 写入 run_id、content_version、触发时间戳
        ↓
[节点6] 并行 HTTP Request（fire-and-forget，三路同时发，互不等待）
  · POST {WF-1 webhook}  body: 全部字段 + run_id + content_version
  · POST {WF-2 webhook}  body: 同上
  · IF 生成视频=是：POST {WF-3 webhook}  body: 同上
  · 每个 HTTP Request 节点：Timeout=10s，Retry On Fail=3 次（仅针对"调不通"，不等业务结果）
        ↓
[节点7] 结束（WF-0 到此返回，全程 < 10s）
```

### 2.3 WF-1 / WF-2 / WF-3 子工作流的统一骨架

⟦实施约束⟧ 三个子工作流遵循同一骨架，保证失败处理一致：

```
[Webhook Trigger] 立即响应 200（Respond Immediately）
        ↓
[主处理链] 各自的业务节点（见第三章 3A/3B/3C 层）
   · 每个外部 API 调用节点：Timeout 显式设置，Retry On Fail=2~3 次，退避等待
   · 长任务（视频/视频）用「轮询模式」，见 2.4
        ↓
[自动检查] 调用检查器（见第三章），产出检查报告
        ↓
[加合规标识] 写入显式+隐式标识（见 2.6）
        ↓
[成功回写] 飞书：对应产物状态=「待审」+ 产物链接 + 检查报告 + 耗时
        ↓
[写运行台账] 追加一条 run 记录（见 2.7）
        ↓
[通知] 飞书消息推送给对应审核人

— 任一节点抛出未捕获异常 → 被 WF-9 错误工作流接住（见 2.5）
```

### 2.4 长任务的轮询模式（视频/慢速图像 API）⟦实施约束⟧

即梦/可灵/部分图像 API 是**异步任务制**：提交返回 task_id，需轮询查状态。子工作流内：

```
[提交任务] POST → 拿到 task_id
        ↓
[Wait 节点] 等待 15~30s
        ↓
[查询状态] GET task_id 状态
        ↓
[IF] 完成？→ 是：取结果继续；否：回到 Wait（设最大轮询次数，如 20 次）
        ↓
[超时保护] 超过最大轮询次数 → 标记该产物「生成超时失败」+ 回写 + 告警
```
> 注：n8n 的 Wait 节点 < 65s 走内存挂起；更长间隔会落库后恢复。视频场景按 15~30s 间隔、上限 ~10min 配置。

### 2.5 失败 / 重试 / 幂等规范（v2.0 缺口补齐）⟦实施约束⟧

| 维度 | 规范 |
|------|------|
| **API 重试** | 每个外部调用节点开启 `Retry On Fail`：最多 3 次，指数退避（如 5s/15s/45s）。仅对"调不通/5xx/超时"重试，对"明确业务拒绝（4xx 参数错）"不重试。 |
| **部分成功** | 三个产物互相独立。海报成功、推文失败 → 海报照常进「待审」，推文状态置「失败」并附原因，互不影响。绝不因一个失败回滚另两个。 |
| **超时** | 每路设硬上限（海报 5min / 推文 5min / 视频 12min）。超时 → 状态「超时失败」+ 告警，不无限挂起。 |
| **幂等键** | 以 `record_id + content_version` 为幂等键。WF-0 节点3 拦截重复触发。子工作流回写时带 content_version，旧版本回写被丢弃（防止迟到的旧任务覆盖新结果）。 |
| **未捕获异常** | 由 `WF-9 错误工作流`（Error Trigger）统一接住：解析出 run_id/产物类型 → 回写对应状态「系统错误」+ 错误堆栈摘要 → 飞书告警工具封装者。 |
| **可重跑** | 任一产物失败后，运营/工具封装者可对该行单独触发"重跑该产物"（仅重发对应子工作流的 webhook，不动其余）。 |

### 2.6 合规标识：生成阶段主动写入（GB 45438-2025）⟦实施约束⟧

依据《人工智能生成合成内容标识办法》及强标 **GB 45438-2025**，AI 生成内容须**同时**具备显式标识与隐式标识，二者并存互补、不可二选一。**本流程在生成阶段主动添加，而非事后检查。**

**显式标识（画面/文本可见）**——制作端添加：

| 产物 | 位置 | 内容 |
|------|------|------|
| 海报 | 图片底边或角处的显著位置（⟦约束⟧ 不与右下角 Logo 重叠，建议底边居中或左下角） | 文字"AI生成"（须含"AI/人工智能"要素 + "生成/合成"要素） |
| 推文 | 正文末尾单独一行 | "本内容由 AI 辅助生成" |
| 视频 | 片尾持续显示 + 画面底部可持续显示文字标识 | "AI生成" |

**隐式标识（文件元数据）**——五要素，制作端写 1–3，传播端补 4–5：

| # | 元数据要素 | 本流程填值 |
|---|-----------|-----------|
| 1 | 生成合成标签 | `AIGC` / `AI生成` |
| 2 | 生成合成服务提供者 | 实际生成服务名（如 `通义万相` / `即梦`）+ 本社区主体名 |
| 3 | 内容制作编号 | 本流程的 `run_id` |
| 4 | 内容传播服务提供者 | （发布到微信/B站时由平台补） |
| 5 | 内容传播编号 | （由传播平台补） |

> ⟦实施约束⟧ 写入方式：图片用 PNG `iTXt`/XMP 块，视频用 MP4 metadata，文本用随附元数据/发布平台字段。格式遵循 GB 45438-2025 附录 E。制作端只负责 1–3，4–5 留空待传播平台填。
> 检查项 `IMG-COMPLY-001 / TXT-COMPLY-001 / VID-COMPLY-001` 由"是否有标识"退化为"**验证显式+隐式标识是否已正确写入**"。

### 2.7 运行台账与可观测性（v2.0 缺口补齐）⟦实施约束⟧

每个子工作流结束（成功或失败）追加一条记录到「运行台账」（飞书独立表格或数据库表），字段：

```
run_id, record_id, content_version, 活动名称, 产物类型(海报/推文/视频),
触发时间, 完成时间, 耗时秒, 结果(成功/失败/超时),
检查_阻断数, 检查_警告数, 检查_通过数,
人工审核决策(通过/修改/驳回，审核后回填), 是否一次通过, 备注
```

用途：第四章「每月 Review」据此统计 **一次通过率趋势 / 各检查项触发率 / 误报率 / 端到端耗时分布**。没有台账，Review 就是拍脑袋。

## 三、逐层拆解

### 第 1 层：运营填表 —— 需求标准化入口

**执行者**：运营人员  **耗时**：~1 分钟

**表单字段定义**（飞书多维表格/表单）：

| 字段名 | 类型 | 必填 | 示例值 | 用途 |
|--------|------|------|--------|------|
| 活动名称 | 单行文本 | ✅ | 开源之夏 2026 | 海报标题、推文标题、视频标题卡 |
| 活动时间 | 日期时间 | ✅ | 2026-07-15 14:00 | 海报日期、推文正文、视频字幕 |
| 一句话简介 | 单行文本 | ✅ | 面向高校学生的开源项目实习计划 | 海报副标题、推文摘要、视频简介卡 |
| 活动详情 | 多行文本 | ✅ | 本届开源之夏共设 200+ 项目... | 推文正文素材、视频旁白文案来源 |
| 视觉风格 | 下拉选择 | ✅ | 科技蓝 / 开源绿 / 社区红 | AI 生成底图的风格方向 |
| 海报模板 | 下拉选择 | ✅ | 活动预告 / 活动回顾 / 嘉宾海报 | 品牌后处理的版式模板选择 |
| 推文类型 | 下拉选择 | ✅ | 活动招募 / 活动回顾 / 嘉宾访谈 | Dify prompt 模板选择 |
| 生成视频 | 单选 | ❌ | 是 / 否 | 是否启用视频生成（默认否） |
| 视频类型 | 下拉选择 | 仅「生成视频=是」时必填 | 活动预热短视频 / 嘉宾金句切片 / 科普动画 | 视频生成模板选择 |
| 视频时长 | 下拉选择 | 仅「生成视频=是」时必填 | 15秒 / 30秒 / 60秒 | 视频时长规格 |
| 视频旁白风格 | 下拉选择 | 仅「生成视频=是」时必填 | 专业沉稳 / 活力轻快 / 科技未来感 | TTS 音色 + BGM 风格 |
| 涉及地图 | 单选 | ❌ | 是 / 否 | 如是，触发地图合规强制人工审核 |
| 特殊要求 | 多行文本 | ❌ | 海报需加合作伙伴 Logo | 透传给审核人 |
| **再生成意见** | 多行文本 | ❌（再生成时用） | 配色换成更冷的蓝 / Tone 再活泼些 | 第 3.8 节再生成回环，回灌进 prompt |
| **（系统）海报状态** | 单选 | 系统写 | 生成中/待审/失败/已定稿 | 看板 |
| **（系统）推文状态** | 单选 | 系统写 | 生成中/待审/失败/已定稿 | 看板 |
| **（系统）视频状态** | 单选 | 系统写 | 生成中/待审/失败/不适用/已定稿 | 看板 |
| **（系统）run_id** | 单行文本 | 系统写 | os2026-1718...| 全链路追踪 |

**设计要点**：
- 运营手填字段控制在 5–7 个必填，视频字段仅在启用时出现；系统字段由 n8n 回写
- 「涉及地图」字段触发审核层的地图合规强制人工审核
- 表单本身就是「需求文档」——不再需要运营额外写需求说明

### 第 3A 层：AI 生成底图（只管创意氛围）

**执行者**：通义万相 / 文心一格 / ComfyUI（按团队选型）  **耗时**：~1 分钟

**核心原则**：AI 只负责出背景氛围图，不管文字、Logo、版式。这些交给后处理。

**Prompt 模板示例**：
```
你是一个品牌视觉生成器。请生成一张背景图：
要求：
- 风格：{视觉风格}（如"科技蓝——深蓝到亮蓝渐变，几何线条，未来感"）
- 构图：抽象背景，无文字，无Logo，无人物
- 色调：{品牌主色调引导词}，深色区域留出顶部和底部（供后续叠加文字层）
- 用途：活动海报背景
严格禁止：任何文字、任何Logo、任何人脸、任何产品图
```

**色彩策略（v3.0 修正）** ⟦实施约束⟧：
- ❌ 不做"事后强行 remap 到品牌色板"——强行映射易把图弄脏弄灰，破坏美感
- ✅ 在**生成端约束**：prompt 内写入品牌主色调引导词；如选型支持，可附风格参考图/固定 seed
- ✅ 后处理只**校验不强改**：采样底图主色，若偏离品牌色板 > 阈值 → 标 🟡 警告交人工判断（`IMG-BRAND-003`），不自动改色
- 若某风格底图反复跑偏，优化 prompt 或在 Phase 0 决策是否引入 LoRA（见第七章决策）

**关键点**：
- ⚠️ Prompt 是主要调优变量——底图风格是否对味取决于 prompt 质量
- ⚠️ 此步不涉及品牌元素——品牌元素全部在后处理中固化
- ⚠️ 如用商业模型 API（通义万相/文心一格），国内合规天然满足；其元数据隐式标识由服务方提供，本流程仍需补齐五要素

### 第 3A-post 层：品牌后处理（稿定可视化模板，低代码）

**执行者**：稿定开放平台「模板设计 API」+ n8n 一个 HTTP 节点 + 一处隐式元数据写入  **耗时**：~30 秒

**核心原则**：品牌元素的叠加由**稿定可视化模板**确定性承载，不依赖 AI 概率，也不依赖手写渲染脚本。**设计师在稿定图形界面里维护模板（Logo 位置、字体、版式、装饰、AI 标识层全部可视化摆放），改品牌零代码**；系统只通过 autofill 接口把活动数据填进模板的变量槽位。

**为什么这样选**（v3.1 决策）：团队代码开发经验少，手写 Pillow/ImageMagick 脚本意味着每次改品牌/加版式都要找人改代码——长期负担。稿定是国内可视化设计 SaaS（设计师熟悉），模板在界面里建、变量在界面里标，渲染零代码。海报是公开营销物料、不含用户隐私，数据过第三方服务的合规敏感度低，故 SaaS 路线可接受。

**处理流水线**（每步标注是否需代码）：
```
输入：AI 底图（背景图变量） + 活动表单数据（文本变量） + 稿定模板ID
  ↓ 步骤1【设计师在稿定预先建好模板，含】：
        · 背景图槽位（填 AI 底图）
        · 标题/日期/副标题 文本槽位（字体/字号/位置/颜色在模板里固定）
        · Logo 图层（固定，思源黑体；右下角）
        · 装饰层（分割线/角标，固定）
        · 二维码图槽位（可选，填动态生成的二维码）
        · 合作伙伴Logo 图槽位（可选）
        · ⟦合规⟧ AI显式标识文本层"AI生成"（固定在底边，不压 Logo）← 模板里摆好，零代码
  ↓ 步骤2【n8n，零代码/低代码】生成二维码图：用 n8n QR 节点或二维码 API，得到图片URL
  ↓ 步骤3【n8n，1个 HTTP 节点】调用稿定模板设计 API：
        POST 稿定API  body: { template_id, 变量映射: {
          bg_image: {AI底图URL}, title: {活动名称}, date: {格式化时间},
          subtitle: {一句话简介}, qr: {二维码URL}, partner_logo: {可选}
        }}
        ↓ 返回渲染好的海报图 URL（稿定异步任务则走 2.4 轮询模式）
  ↓ 步骤4【n8n，残留唯一代码点·一次性配置】写入隐式标识元数据：
        用 n8n Code/Execute Command 节点，对返回图片注入 PNG iTXt/XMP 五要素1–3（见 2.6）
        ← 这是整条管线唯一需要少量代码的地方，工具封装者配一次，之后不动
  ↓ 步骤5【n8n】色彩校验（可选，仅采样比对，超阈值标 🟡 警告，不强改）
  ↓
输出：品牌一致的海报终稿初版 + 生成元数据 JSON（含 稿定模板ID、run_id）
```

> ⟦实施约束⟧ 品牌一致性的"真相源"是**稿定模板**本身，不再是 brand_config.json 里的坐标/字体/字号。设计师改版式 = 在稿定界面改模板，无需任何人改代码或 JSON。

**精简后的品牌配置**（`brand_config.json`，仅保留稿定 autofill 管不到的部分，git 版本管理）：

```json
    "event_preview": {"template_id": "稿定模板ID-xxx", "description": "活动预告海报",
      "fields": ["bg_image","title","date","subtitle","qr"]},
```

> 字体（思源黑体）、Logo、装饰、文字坐标/字号/颜色、AI 标识层位置——**这些全部内化进稿定模板**，不再出现在 JSON 里。JSON 只剩"模板ID映射 + 校验用色板 + 标识文案"。

**为什么这条路兼顾「可控」与「低代码」**：

| | LoRA 方案 | 手写脚本方案(旧) | 稿定模板方案(v3.1) |
|------|------|------|------|
| 字体/Logo/版式一致性 | 依赖 AI 渲染，可能错 | 代码写死，一致 | 模板固定，一致 |
| 日期准确性 | AI 可能渲染错 | 代码格式化 | 变量填充，不会错 |
| **改品牌由谁做** | 重新训练 LoRA（工程） | 工具封装者改代码 | **设计师改模板（零代码）** |
| **加新版式** | 重训 | 改代码 | 设计师在稿定新建模板 |
| 残留代码量 | 高 | 高 | **仅"写隐式元数据"一处，配一次** |
| 成本 | GPU 算力 | 零订阅但高人力 | 稿定企业 API 订阅 |

## 三续、内容审核体系（可扩展）

### 3.1 设计原则

**双层审核**：自动检查（机器）→ 人工审核（人）。

**分级原则（v3.0 重校准）** ⟦实施约束⟧：
- 🔴 **阻断（自动驳回）**：**仅限确定性、高精度**检查——正则、白名单/黑名单匹配、HTTP 404、元数据字段存在性、模板位置校验。
- 🟡 **警告/强制人审**：**所有概率性检测器**——图像分类（地图）、敏感图像 API、LLM 打分、OCR 比对、相似度。这类即使疑似命中也**不自动驳回**，而是标记并**强制进入人工复核**。
- 🟢 **通过**：机器检查无问题，正常进入人工审核。

> 理由：概率检测器误报会把好内容自动打回运营，造成高摩擦回环；而漏报（尤其地图）后果严重。故概率项一律转人工，把"决定权"交给人，机器只做"提请注意"。

**地图安全网** ⟦实施约束⟧：以**表单「涉及地图=是」为权威触发器**，强制人工审核；图像分类检测器仅作 best-effort 兜底（🟡），不作为唯一闸门。

**可扩展**：检查规则按内容类型（图片/文案/视频）和维度（品牌/文字/合规/其他）注册，新增规则只需加检查函数 + 注册一行 JSON。

**检查结果分级**：

| 级别 | 含义 | 行为 |
|------|------|------|
| 🔴 阻断 | 确定性违规 | 自动驳回并附修改建议，不进人工 |
| 🟡 警告/强制人审 | 概率疑似 | 标记在报告，强制人工复核 |
| 🟢 通过 | 机器无问题 | 正常进入人工审核 |

```json
  "category": "品牌一致性",
  "applies_to": ["图片"],
  "severity": "blocking",
  "deterministic": true,
  "description": "检查海报中是否包含社区Logo",
  "check_function": "check_logo_presence",
  "enabled": true, "added": "2026-06-16", "added_by": "设计师"

```
> ⟦实施约束⟧ 新增字段 `deterministic`(bool)。校验规则：`severity=blocking` 必须 `deterministic=true`；`deterministic=false` 的规则 severity 只能是 `warning`。注册时校验该约束。

**规则管理**：配置文件 `review_rules.json`，由工具封装者维护。

### 3.6 自动检查执行流程

```
检查器（生成完成后、推送审核前）
  Step1 读 review_rules.json → 加载启用规则（并校验 deterministic 约束）
  Step2 按产物类型 + 表单标志筛选规则（涉及地图=是 → 强制加载地图类）
  Step3 逐个执行，生成结果 [{rule, result(pass/warn/fail), detail, suggestion}]
  Step4 生成检查报告：阻断N/警告M/通过K，🔴优先展示，每项附建议
  Step5 路由：
        · 有 🔴 阻断 → 自动驳回，飞书通知修正
        · 仅 🟡/🟢 → 推人工审核，报告作参考（🟡 项强制审核人逐条确认）
```

### 3.7 审核执行与复核流程

```
审核人收到飞书通知 → 打开审核包（产物预览 + 检查报告 + 原始需求）
  ├─ 🔴（已自动驳回）→ 确认修改是否到位
  ├─ 🟡 强制人审 → 逐条判定接受/修改（地图项必须对照官方地图逐项核对）
  └─ 🟢 → 快速过
人工补充判断（机器覆盖不了的）：视觉审美 / 社区语境 / 内容判断 / 策略方向
审核决策：
  ├─ 通过 → 状态置「已定稿」，进入发布流程，回填台账决策=通过/一次通过
  ├─ 修改 → 走第 3.8 节再生成回环
  └─ 驳回 → 填原因 → 运营调整需求后重新提交
```

### 3.8 人工反馈 → 再生成回环（v2.0 缺口补齐）⟦实施约束⟧

审核人的修改意见常是自然语言（"换个配色""Tone 更活泼""CTA 放开头"），这些不是结构化字段。回灌路径：

```
审核人点击飞书「重新生成-海报 / 重新生成-推文」按钮（飞书按钮字段/自动化）
  ↓ 在「再生成意见」多行文本字段填写自然语言意见
  ↓ 触发 WF-0（带 regen=true 标志）
  ↓ WF-0 节点2：content_version += 1（幂等键更新，旧任务回写将被丢弃）
  ↓ WF-0 仅重发对应产物的子工作流 webhook（不动其余产物）
  ↓ 子工作流将「再生成意见」拼接进 prompt：
      · 海报：追加到底图生成 prompt（"额外要求：{再生成意见}"）
      · 推文：追加到 Dify 输入（"修改要求：{再生成意见}，保留其余不变"）
  ↓ 生成 → 检查 → 回写「待审」（content_version+1）→ 再次人工审核
```
- 设最大再生成次数（如 3 次），超过则提示转人工手动处理，避免无限回环
- 每次再生成在台账留痕（content_version 区分），便于复盘哪些意见反复出现 → 沉淀进 prompt/模板

## 七、Phase 0 先行验证（第 1-2 周）

### 验证目标
走通「一张表 → AI 底图 → 后处理(含标识) → 自动检查 → 人工审核 → 海报定稿」+「推文定稿」最简流程，**且验证异步/失败/再生成机制可用**。

### 最简范围

| 维度 | 最简范围 |
|------|----------|
| 海报模板 | 1 种（活动预告），1 种风格（科技蓝） |
| 后处理 | 稿定 1 个模板（Logo+标题+日期+分割线+**AI标识层**）+ autofill 填充跑通 |
| 推文模板 | 1 种（活动招募） |
| AI 底图 | 商业模型 API，先验证 prompt 效果 |
| 检查规则 | 核心阻断项（Logo/日期/链接/AI标识）+ 地图强制人审，🟡 项逐步加 |
| 编排 | WF-0 + WF-1 + WF-2 + WF-9，**异步 webhook + 幂等 + 失败回写必须跑通** |
| 再生成 | 至少验证 1 次「海报再生成意见回灌」 |
| 视频 | 不启用 |
| 台账 | 启用，记录每次 run |
| Dify / n8n | 社区版自部署（PostgreSQL + 公网 HTTPS 入口，详见第十章） |

### 验证成功标准

| # | 指标 | 通过线 |
|---|------|--------|
| G1 | 设计师海报一次通过率 | ≥ 60%（底图对味 + 品牌元素正确 + 标识正确） |
| G2 | 运营推文改稿幅度 | < 40%（初稿可保留 60%+ 正文） |
| G3 | 端到端耗时 | < **P0 基线**人工耗时的 70% |
| G4 | 检查准确率 | 🔴 阻断项无漏报 + 🔴 误报率 < 10%；🟡 项无致命漏报（地图必拦住） |
| **G5** | **健壮性** | 单产物失败不影响其余；重复触发不重复生成；再生成回环可用 |

### 验证决策
- ✅ 全过 → 启动 Phase 1
- ⚠️ G1 不达标 → 优化 prompt 或考虑 LoRA
- ⚠️ G2 不达标 → 换模型/优化 prompt/扩 RAG
- ⚠️ G4 不达标 → 调检查阈值或重新归类 severity
- ⚠️ G5 不达标 → 编排架构（异步/幂等/错误工作流）返工，**这是硬阻断，不可带病进 Phase 1**
- ❌ 全未通过 → 重新评估方案

## 八、新增检查规则的流程（可扩展示例）

**场景**：发现海报用了旧版 Logo，现有规则没抓到。

```
Step1 设计师提需求："检查 Logo 是否最新版本"
Step2 工具封装者写函数 check_logo_version()：裁 Logo 区→与标准 Logo 算 SSIM→<0.95 标记
Step3 注册 review_rules.json（注意 deterministic 字段与 severity 约束）：
      {"id":"IMG-BRAND-005","category":"品牌一致性","applies_to":["图片"],
       "severity":"warning","deterministic":false,
       "description":"检查海报Logo是否为最新版本","check_function":"check_logo_version",
       "enabled":true,"added":"2026-07-20","added_by":"设计师"}
      （SSIM 为概率相似度 → deterministic=false → 只能 warning，符合约束）
Step4 测试 → 上线，下次自动检查生效
```

## 九、风险提示

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 工具封装者角色空缺 | **致命** — 流程无法启动 | Phase 0 最高优先级确认 |

| 合规标识缺失 | **法律风险**（GB 45438 强标） | 生成阶段主动写显式+隐式标识；🔴 校验存在性 |
| 字体/BGM 版权 | 法律风险 | 稿定模板内用思源黑体（开源可商用）+ BGM 授权白名单 |
| 稿定 API 不可用/限流/涨价 | 海报管线中断 | 配额监控告警；模板可导出留档；保留"必要时回退手写脚本"的应急路径 |
| AI 视频文字渲染错误 | 信息错误传播 | 视频关键文字强制 OCR + 人审 |
| LLM Hallucination | 事实错误 | 核查报告 + 人工审核双保险 |
| 检查规则覆盖不全 | 未定义风险漏过 | 每月据台账 Review 回顾漏检 → 新增规则 |
| **飞书回调打不进内网 n8n** | **致命** — 整条流程无法触发 | P15 必须先备好公网 HTTPS 入口（域名+证书+反向代理）；详见 10.1 |
| **n8n 单进程模式 + Wait/长任务** | 重启丢执行、并发阻塞 | PostgreSQL 持久化 + 视并发启用 queue 模式（Redis）；详见 10.3 |

| 自部署运维（升级/备份/宕机） | 长期负担 | PostgreSQL 定时备份 + N8N_ENCRYPTION_KEY 留存 + 健康检查告警；详见 10.9 |

## 十、自部署与实施细则（交付自部署 Agent 的工程规格）

> 本章补齐"自己开发 + 自己部署"所需的基础设施、集成凭证、服务间契约、网络与测试细节。前九章讲"逻辑怎么流"，本章讲"机器上怎么搭、怎么连、怎么验"。标 `⟦实施约束⟧` 为硬性要求。

### 10.1 部署拓扑与网络（先解决这个，否则全盘不通）

```
        公网
          │  HTTPS(443) 仅暴露 n8n 的 webhook 路径
          ▼
   [反向代理 Nginx/Caddy] ── TLS 证书(Let's Encrypt/企业证书)
          │ 内网
   ┌──────┼───────────────────────────────────────────┐
   │      ▼                                             │
   │   [n8n]  ──→ [PostgreSQL] (n8n执行数据/Wait持久化)  │
   │     │    ──→ [Redis]      (queue模式,可选)          │
   │     │                                              │
   │     ├─出站→ 稿定API / AI图像API / Dify / 云内容安全API / 飞书OpenAPI
   │     ▼                                              │
   │   [Dify] ──→ [向量库/PostgreSQL/Redis]              │
   └────────────────────────────────────────────────────┘
   飞书(SaaS) ──事件/自动化 HTTP POST──→ 公网HTTPS入口 ──→ n8n webhook
```

⟦实施约束⟧ **关键坑**：飞书是公有云，它要把"新增记录"事件 POST 给 n8n，**n8n 的 webhook 必须有公网可达的 HTTPS 地址**。若 n8n 只在内网，飞书回调进不来，整条链路无法触发。落地必须先具备：①固定域名；②有效 TLS 证书；③反向代理把 `https://<域名>/webhook/...` 转发到 n8n。出站方向（n8n→各API）走内网网关即可。

⟦实施约束⟧ 仅暴露 n8n 的 webhook 路径到公网；n8n 管理界面、Dify、数据库一律不暴露公网，只内网/VPN 访问。

### 10.2 服务器规格与部署清单

| 组件 | 最低规格 | 部署方式 | 持久化卷 |
|------|---------|---------|---------|
| n8n + PostgreSQL(+Redis) | 2 vCPU / 4GB / 40GB（Phase 0）；生产 4 vCPU / 8GB | Docker Compose，镜像版本**固定 pin**（如 `n8nio/n8n:1.x.x`），勿用 `latest` | n8n data、PostgreSQL data |
| Dify 社区版 | 2 vCPU / 4GB / 40GB | 官方 docker-compose（含 api/worker/web/db/redis/向量库） | DB、向量库 |
| 共享存储 | — | brand-assets 目录（git 仓库）；产物临时存储（对象存储/本地卷） | git repo、产物 |

⟦实施约束⟧ 所有镜像**固定版本号**；`docker-compose.yml`、`.env.example`、Nginx 配置纳入 git（`.env` 真值不入 git）。

### 10.3 n8n 生产配置（环境变量清单）⟦实施约束⟧

| 变量 | 值/说明 | 为什么 |
|------|---------|--------|
| `DB_TYPE` / `DB_POSTGRESDB_*` | postgres + 连接信息 | **默认 SQLite 不抗并发、Wait 恢复不可靠**，生产必须 PostgreSQL |
| `EXECUTIONS_MODE` | `queue`（并发高时）+ Redis；Phase 0 可 `regular` | 异步子工作流并发 + Wait 节点恢复需要 |
| `N8N_ENCRYPTION_KEY` | 固定随机串，**务必备份** | 凭证加密密钥；丢了所有 credentials 失效、无法迁移 |
| `WEBHOOK_URL` | `https://<公网域名>/` | 决定飞书回调和子工作流互调用的 webhook 地址正确 |
| `N8N_HOST` / `N8N_PROTOCOL` | 域名 / https | 同上 |
| `GENERIC_TIMEZONE` | `Asia/Shanghai` | 日期格式化、定时正确 |
| `EXECUTIONS_DATA_PRUNE` / `_MAX_AGE` | true / 如 336（小时） | 防执行数据撑爆数据库 |
| `N8N_BASIC_AUTH_*` 或反代鉴权 | 管理界面登录保护 | 防未授权访问编排后台 |

- **凭证管理**：飞书/稿定/AI图像/Dify/云安全 的 key 全部存入 **n8n Credentials**（加密），节点引用凭证而非硬编码。
- **子工作流 webhook 鉴权**：WF-1/2/3 的 Webhook 节点校验自定义 header（如 `X-Internal-Token`），WF-0 调用时带上，防外部伪造触发（见 10.7）。

### 10.4 飞书自建应用配置 ⟦实施约束⟧

1. **创建自建应用**，获取 `app_id` / `app_secret`。
2. **权限范围(scope)**：多维表格读写（`bitable:app`）、发消息/机器人（`im:message`、`im:message:send_as_bot`）、按需文件/图片上传权限。
3. **触发方式（二选一，推荐 A）**：
   - **A. 多维表格自动化（推荐，最简）**：在多维表格里配「自动化」——*当记录被创建* → 动作*发送 HTTP 请求* → POST 到 n8n 的 WF-0 webhook（带表内字段 + 内部鉴权 header）。无需处理事件加密。
   - **B. 事件订阅**：订阅多维表格记录变更事件 → 回调到 n8n。需处理 **URL challenge 验证** + **Encrypt Key 解密** + **Verification Token 校验**，较复杂。
4. **回写与通知用 OpenAPI**：先用 `app_id/app_secret` 换 **`tenant_access_token`（有效期约 2h，需缓存+到期刷新）**，再调用多维表格"更新记录"和"发送消息卡片"接口。
5. **审核交互**：用飞书**消息互动卡片**（含预览缩略图 + 深链到记录 + 可选「通过/驳回」按钮，回调到 n8n 另一个 webhook）。

### 10.5 密钥与配置清单（一处汇总，便于 Agent 落地）⟦实施约束⟧

| 密钥/配置 | 用途 | 存放 |
|-----------|------|------|
| 飞书 app_id / app_secret | 换 token、读写多维表格、发消息 | n8n 凭证 / .env |
| 飞书 内部 webhook 鉴权 token | 校验回调来源 | n8n 凭证 / .env |
| 稿定 API key/secret | 模板 autofill 渲染 | n8n 凭证 |
| AI 图像 API key（通义万相/文心一格等） | 出底图 | n8n 凭证 |
| Dify API key | 推文生成 | n8n 凭证 |
| 云内容安全 API key（阿里/腾讯） | 敏感图/敏感内容检测 | n8n 凭证 |
| （可选）即梦/可灵 API key | 视频 | n8n 凭证 |
| `N8N_ENCRYPTION_KEY` / PostgreSQL 密码 | n8n 自身 | .env（备份留存） |

⟦约束⟧ `.env`、私钥、证书全部 `.gitignore`；仓库只放 `.env.example`（占位）。

### 10.6 服务间数据契约（Agent 按所选服务商 API 补实）

> 以下给出**字段骨架与必备项**；具体 endpoint/字段名以最终选定服务商的 API 文档为准，落地时补实并写进 n8n 节点。

- **WF-0 → 子工作流 webhook（body）**：
  ```json
  { "run_id":"...", "record_id":"...", "content_version":1, "regen":false,
    "fields": { "活动名称":"...", "活动时间":"2026-07-15T14:00:00+08:00", "一句话简介":"...",
                "活动详情":"...", "视觉风格":"科技蓝", "海报模板":"event_preview",
                "推文类型":"活动招募", "涉及地图":false, "特殊要求":"...", "再生成意见":"" },
    "_auth":"<X-Internal-Token>" }
  ```
- **稿定 模板设计 API（请求要点）**：`template_id` + 变量映射（`bg_image` 用 AI 底图 URL，`title/date/subtitle/qr` 等文本/图片变量）；响应取**渲染图 URL**；若异步则按 2.4 轮询。
- **AI 图像 API**：prompt（含色调引导词）+ 尺寸（1080×1920）+ 风格/seed（如支持）；响应取**底图 URL**。
- **Dify API**：用 workflow/对话 API，`inputs` 传 {活动名称/时间/简介/详情/推文类型/再生成意见}；响应取**推文正文 + 标题备选 + 核查标注**。
- **云内容安全 API**：传图片/文本 URL 或内容；响应取**风险标签**（命中 → 🟡 强制人审）。
- **飞书 更新记录 / 发卡片**：record_id + 字段值（状态/产物URL/检查报告）；卡片 JSON（标题/图片/按钮+回调）。
- **检查报告（统一结构）**：`[{ "rule":"IMG-MAP-001", "result":"pass|warn|fail", "detail":"...", "suggestion":"..." }]` + 汇总 `{blocking:N, warning:M, pass:K}`。

### 10.7 安全基线 ⟦实施约束⟧

- 公网只暴露 n8n webhook 路径；管理台/数据库/Dify 不出内网。
- 子工作流与卡片回调 webhook **校验内部 header token**，拒绝无 token 请求。
- 若用飞书事件订阅，**校验 Verification Token + 解密 Encrypt Key**。
- 全部密钥走 n8n 凭证/.env，不入 git；定期轮换。
- 出站调用使用 HTTPS。

### 10.8 测试与验收（组件级 → 端到端）

**组件冒烟（逐个先单测）**：
1. 飞书自动化能 POST 到 n8n webhook（先用一个回显工作流验证打通公网）。
2. n8n 能用 tenant_access_token 回写多维表格一条字段 + 发一张卡片。
3. 稿定 API 用固定模板 + 假数据渲染出图。
4. AI 图像 API 出一张底图。
5. Dify API 生成一段推文。
6. 隐式元数据写入：渲染图注入五要素后，能读回校验。

**端到端用例（必须全过 = Phase 0 G5）**：
- E1 Happy path：填一行 → 海报+推文都到「待审」→ 审核通过 → 「已定稿」。
- E2 部分失败注入：临时让推文 API 失败 → 海报仍「待审」，推文「失败」附原因，互不影响。
- E3 重复触发：同一行重复触发 → 幂等拦截，不重复生成。
- E4 再生成：填「再生成意见」点重新生成 → content_version+1，仅海报重出，旧任务回写被丢弃。
- E5 阻断项：构造日期格式错 → 🔴 自动驳回 + 通知。
- E6 强制人审：涉及地图=是 → 必进人工，未自动放行。
- E7 超时：模拟视频/图像超时 → 标「超时失败」+ 告警，不挂死。

### 10.9 发布与运维

- **定稿后发布**：状态置「已定稿」后，运营从飞书记录下载海报图/复制推文，**手动发布**到微信公众号/B站等（含传播端补隐式标识 4–5 要素、平台AI声明）。本流程到「定稿」为止，不直接对接发布渠道（如需自动发布为后续增强）。
- **监控告警**：WF-9 错误工作流 + 各超时/失败分支统一推**飞书告警群**（机器人 webhook）；n8n/Dify 容器健康检查。
- **备份**：PostgreSQL 定时备份；`N8N_ENCRYPTION_KEY` 异地留存；brand-assets git 远程仓库。
- **升级**：n8n/Dify 升级前备份 + 在测试实例验证子工作流不被破坏，再灰度。
