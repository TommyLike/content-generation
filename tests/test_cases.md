## A. 组件冒烟测试（每次部署后逐个执行）

### A1 — 飞书 webhook 可达性

- [ ] 在飞书多维表格新增一行测试记录
- [ ] 观察 n8n WF-0 执行日志，确认 webhook 被触发
- [ ] 确认 WF-0 返回 200
- [ ] **通过标准**: WF-0 执行成功，回写状态为「生成中」

### A2 — n8n Token 刷新 + 飞书回写

- [ ] 用 Postman/curl 模拟 WF-1 回写请求
- [ ] 确认飞书多维表格对应记录状态字段更新
- [ ] **通过标准**: 记录字段更新正确，无 401/403

### A3 — 稿定模板渲染

- [ ] 用固定测试底图 URL + 测试文本
- [ ] 调用稿定 autofill API
- [ ] 检查返回图片是否包含所有元素
- [ ] **通过标准**: 图片 URL 返回，Logo/文字/日期/AI标识 全部可见

### A4 — AI 底图生成

- [ ] 用测试 prompt 调用通义万相 API
- [ ] 检查返回图片尺寸 = 1080×1920
- [ ] **通过标准**: 图片生成成功，无明显异常

### A5 — Dify 推文生成

- [ ] 用测试活动数据调用 Dify workflow API
- [ ] 检查返回含标题备选、正文、核查标注
- [ ] **通过标准**: 三要素齐全，正文 > 100 字

### A6 — 隐式元数据写入/读取

- [ ] 对测试图片执行 write_metadata.py
- [ ] 执行 check_metadata.py 验证
- [ ] **通过标准**: 元数据五要素 1-3 已写入且可读回

## B. 端到端测试（Phase 0 G5 必过）

### E1 — Happy Path（全链路通过）

**前置条件**: 所有服务正常

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 运营在飞书表格新增一行测试活动 | WF-0 触发 |
| 2 | 等待 ~3min | 海报状态 → 「待审」，推文状态 → 「待审」 |
| 3 | 审核人点击卡片「通过」 | 状态 → 「已定稿」 |
| 4 | 检查台账 | 两条记录（海报+推文），结果=成功 |

**通过标准**: 无阻断、无异常、台账正确

---

### E2 — 部分失败注入

**前置条件**: 手动让推文 API 不可用（如临时改错 Dify API Key）

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 新增一行 | WF-0 触发 |
| 2 | 等待 ~3min | 海报状态 → 「待审」；推文状态 → 「失败」附原因 |
| 3 | 检查 | 海报不受影响，推文展示了错误原因 |

**通过标准**: 一个产物失败不拖垮另一个

---

### E3 — 重复触发拦截（幂等）

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 对同一行（已处于「待审」状态）再次触发 | WF-0 节点3 拦截 |
| 2 | 检查飞书消息 | 收到 "该行已在处理中" 提示 |
| 3 | 检查台账 | 无重复 run 记录 |

**通过标准**: 重复触发被拦截，不产生重复产物

---

### E4 — 再生成回环

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 正常生成一次 | 海报 V1「待审」 |
| 2 | 审核人填「再生成意见」= "换成更冷的蓝色" |
| 3 | 触发「重新生成-海报」 | WF-0 content_version=2 |
| 4 | 等待 | 新海报 V2「待审」，旧版本回写被丢弃 |
| 5 | 检查台账 | 两条记录：V1(content_version=1) + V2(content_version=2) |

**通过标准**: content_version 递增、旧版本数据不覆盖新版本

---

### E5 — 阻断项：日期格式错

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 填表，故意把「活动时间」填为 "下周三" | WF-0 参数校验：日期格式不合法 → 状态=「校验失败」 |

或者（如果 WF-0 放过格式检查）：

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 生成完成，进入检查阶段 | IMG-TEXT-002 检测到日期格式不对 |
| 2 | 检查路由 | 🔴 阻断 → 状态=失败 + 飞书通知 |

**通过标准**: 阻断项自动驳回，不进人工

---

### E6 — 强制人审：涉及地图

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 填表，「涉及地图」= 是 | WF-0 正常触发 |
| 2 | 生成完成，进入检查阶段 | IMG-MAP-001~004 全部标记 🟡 |
| 3 | 检查路由 | 无 🔴 → 状态=「待审」，但检查报告标注地图警告 |
| 4 | 审核人收到卡片 | 🟡 强制人审标记突出，地图项需逐条确认 |

**通过标准**: 涉及地图=是 时，即使图片类检测器未检测到地图，也必须强制进入人工审核

---

### E7 — 超时处理

| 步骤 | 操作 | 期望结果 |
|------|------|----------|
| 1 | 模拟视频 API 不响应（如用不存在的 URL） | 轮询 20 次后超时 |
| 2 | 检查 | 状态 → 「超时失败」+ WF-9 告警 |
| 3 | 检查 | n8n 执行不挂死，其他产物不受影响 |

**通过标准**: 超时后失败回写 + 告警，不无限挂起
